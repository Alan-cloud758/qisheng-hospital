# 启胜医院 HIS 平台扩展设计

## 背景

启胜医院当前已经具备多端门诊医疗平台雏形，包含 Express + TypeScript + Prisma 后端、Vue 管理端、uni-app 患者端，以及完整门诊演示流程。现有能力覆盖预约、签到、医生接诊、病历、诊断、医嘱、处方、模拟收费、药房审核发药、患者端查看预约和就诊记录。

下一阶段目标是把项目从“完整门诊演示系统”推进为“启胜医院 HIS 演示平台”。扩展范围包括门诊运营增强、住院管理、医保模拟结算、LIS 检验和 PACS 影像。

## 系统边界

本项目继续定位为可运行、可演示、可扩展的医院信息系统样例，不直接接入真实外部平台。

包含：

- 管理端主数据运营能力。
- 门诊业务增强。
- 模拟支付、退款和费用闭环。
- 药房库存。
- 诊疗效率和质量控制。
- 患者端提醒、排队、复诊入口。
- 运营看板升级。
- 基础住院管理。
- 医保模拟结算。
- LIS 检验模拟闭环。
- PACS 影像模拟闭环。

不包含：

- 真实微信、支付宝或银行支付渠道。
- 真实医保平台接口。
- 真实 LIS 设备或第三方 LIS 系统。
- 真实 PACS、DICOM 影像服务或影像设备。
- 复杂护理文书、手术麻醉、ICU、病案首页、DRG/DIP。

支付、医保、LIS、PACS 都通过 provider/adapter 层预留外部对接能力，默认实现为项目内 mock provider。

## 系统分层

### 基础运营层

基础运营层提供全系统复用的数据和后台能力：

- 账号、角色、权限、菜单。
- 院区、科室、诊室、病区、床位。
- 医生、患者、就诊人。
- 费用项目、药品目录、公告、数据字典。
- 审计日志。

目标是让管理端从只读演示转为可维护运营后台。

### 门诊业务层

门诊业务层在现有流程上继续增强：

- 排班、号源、预约、签到、改约、停诊、爽约。
- 医生接诊、病历、诊断、医嘱、处方。
- 模拟支付、退款、费用订单和流水。
- 药房库存、审核、发药、退药。
- 患者提醒、排队、收藏医生、复诊入口。
- 门诊运营看板。

目标是把门诊流程打磨成完整、可追踪、可运营的业务闭环。

### 扩展医疗业务层

扩展医疗业务层新增 HIS 关键模块：

- 住院管理。
- 医保模拟结算。
- LIS 检验。
- PACS 影像。

目标是形成 HIS 雏形，同时保持每个模块都能在项目内独立演示。

## 总体实施路线

采用分阶段、端到端交付方式。每个阶段都包含数据库模型、种子数据、后端 API、前端或小程序页面、测试和构建验证。

阶段顺序：

1. 管理端可运营化。
2. 排班和号源增强。
3. 模拟支付/退款闭环。
4. 药房库存。
5. 诊疗效率和质量。
6. 患者端提醒和排队。
7. 运营看板升级。
8. 住院管理。
9. 医保模拟结算。
10. LIS 检验。
11. PACS 影像。

## 数据模型原则

### 共用患者与就诊主体

继续复用 `PatientProfile` 和 `VisitMember`。门诊、住院、医保、检验、影像都关联同一套患者/就诊人数据，避免每个模块重复建患者档案。

### 统一费用中心

现有 `PaymentOrder` 和 `PaymentOrderItem` 扩展为统一费用中心。挂号费、药品费、检验费、检查费、住院费都进入统一订单和明细。

新增支付流水、退款单、退款流水和可选结算批次。后续医保结算也基于统一费用明细做拆分。

### 统一医嘱中心

现有 `MedicalOrder` 后续扩展为门诊和住院可复用的医嘱能力。医嘱类型覆盖药品、检验、检查、治疗、护理等。

LIS 和 PACS 申请由医嘱派生，避免医生端重复录入业务来源。

### 药品与库存复用

现有 `DrugCatalog`、`Prescription`、`PrescriptionItem` 继续作为处方基础。新增库存批次和库存流水。门诊处方和住院处方共用药品目录和库存扣减规则。

### 报告中心

LIS 检验报告和 PACS 影像报告分别建业务申请单，但共享报告发布、查看权限、附件/链接和患者可见性原则。

### 审计与状态流转

关键动作写入 `AuditLog`。状态流转规则放在 service 层，不直接散落在 route 或 Vue 页面中。

## API 设计原则

保留现有分组并扩展：

- `/api/admin/*`：主数据、运营配置、看板、系统管理。
- `/api/staff/doctor/*`：门诊和住院医生工作台。
- `/api/staff/cashier/*`：模拟支付、退款、订单处理。
- `/api/staff/pharmacy/*`：处方审核、发药、库存。
- `/api/mini/*`：患者端预约、提醒、报告、费用、住院信息。
- `/api/staff/nurse/*`：住院入院、床位和基础护理操作。
- `/api/staff/lab/*`：采样、结果录入、检验报告。
- `/api/staff/radiology/*`：影像检查、报告录入、影像链接维护。
- provider service 层：医保、LIS、PACS、支付模拟适配。

接口规则：

- 查询接口支持分页、关键词、状态、日期范围。
- 动作接口使用明确语义，例如 `POST /refund`、`POST /review`、`POST /publish`。
- 状态动作必须走 service 层校验。
- 返回结构保持 `{ item }` 和 `{ items, pagination }` 风格。
- 模拟外部系统统一使用 provider 接口，默认 mock 实现。

## 阶段 1：管理端可运营化

目标：把管理端从只读列表升级为可维护运营后台。

后端范围：

- 为院区、科室、诊室、医生档案、费用项目、药品目录、公告、数据字典、账号、角色、菜单补 CRUD。
- 支持分页、关键词、状态筛选、院区/科室筛选。
- 支持新增、编辑、启用、停用。
- 优先软停用，不做默认物理删除。
- 校验编码、账号、药品编码、科室编码等唯一性。
- 新增、修改、启停用写审计日志。

前端范围：

- 列表顶部搜索表单。
- 状态、科室、院区筛选。
- 分页表格。
- 新增/编辑弹窗或抽屉。
- 启用/停用确认。
- 保存失败提示和字段校验。
- 字典项在分类详情中维护。

不先做复杂万能表单引擎。可以先做轻量配置式列表组件，复杂资源保留专门页面。

验收标准：

- 管理员能维护基础主数据。
- 停用科室、医生、药品不会出现在患者端可选列表中。
- 列表统一分页，不一次性返回全部数据。
- 关键操作进入审计日志。

## 阶段 2：排班和号源增强

目标：让排班/号源支持后台运营、患者改约和状态追踪。

新增或扩展模型：

- `ScheduleTemplate`：排班模板。
- `ScheduleTemplateRule`：模板规则。
- `ScheduleChangeLog`：排班变更日志。
- `AppointmentSlot.lockedUntil`：锁号过期时间。
- `RegistrationChangeLog`：挂号变更历史。
- `Registration.noShowAt`、`noShowReason`：爽约标记。

后端能力：

- 创建和维护排班模板。
- 按模板批量生成未来排班和号源。
- 后台停诊并提示受影响预约。
- 患者改约，释放旧号源并占用新号源。
- 锁号超时释放。
- 爽约标记。
- 按医生、科室、日期统计总号、余号、已约、停诊。

患者端：

- 预约确认页显示锁号倒计时。
- 我的预约提供改约入口。
- 仅允许 `BOOKED` 且未签到的预约改约。

状态规则：

- 号源：`AVAILABLE -> LOCKED -> BOOKED`。
- 锁定超时：`LOCKED -> AVAILABLE`。
- 取消/改约释放：`BOOKED -> AVAILABLE`。
- 挂号：`BOOKED -> CHECKED_IN -> IN_VISIT -> COMPLETED`。
- 爽约：`BOOKED -> NO_SHOW`。

验收标准：

- 管理员能用模板批量生成号源。
- 患者能改约。
- 锁号不会长期占用资源。
- 停诊和爽约都有明确状态和历史记录。

## 阶段 3：模拟支付/退款闭环

目标：保留模拟支付，但补齐订单、流水、退款和审计。

新增模型：

- `PaymentTransaction`：模拟支付流水。
- `RefundOrder`：退款申请。
- `RefundTransaction`：退款流水。
- `PaymentSettlement`：可选结算批次，服务后续医保和住院。

扩展 `PaymentOrder`：

- `businessType`：REGISTRATION、PRESCRIPTION、LIS、PACS、INPATIENT 等。
- `sourceType`、`sourceId`：关联业务来源。
- `cancelledAt`、`refundedAt`、`payMethod`。

后端能力：

- 模拟支付：`PENDING -> PAID`，生成支付流水。
- 未支付订单取消。
- 已支付订单退款申请、审核和执行。
- 订单明细重算。
- 统一费用查询。
- 支付、取消、退款动作写审计日志。

模拟 provider：

- `PaymentProvider.pay(order)`。
- `PaymentProvider.refund(refundOrder)`。
- `PaymentProvider.query(transactionNo)`。

默认 `MockPaymentProvider`，不接真实渠道。

验收标准：

- 收费员能模拟收费、取消订单、退款。
- 支付和退款都有流水。
- 患者端能看到费用状态。
- 费用系统可接入药品、检验、影像、住院费用。

## 阶段 4：药房库存

目标：让药房从处方审核/发药升级为有库存约束的药品管理。

新增模型：

- `DrugStockBatch`：药品批次库存。
- `DrugStockMovement`：库存流水。
- `DrugStockAlert`：库存预警。

扩展 `DrugCatalog`：

- 最小库存阈值。
- 是否需要批次管理。
- 是否停用。

后端能力：

- 药品入库。
- 库存调整和报损。
- 审方时检查药品停用和库存风险。
- 发药时按批次扣减库存，优先近效期。
- 退药生成反向库存流水。
- 查询低库存、近效期、过期预警。

业务规则：

- 停用药品不能新开处方。
- 库存不足不能发药。
- 过期批次不能发药。
- 发药扣库存在事务内完成。
- 库存调整必须记录原因。

验收标准：

- 药房能管理库存。
- 发药受库存约束。
- 每次库存变化有流水。
- 看板能展示低库存和近效期预警。

## 阶段 5：诊疗效率和质量

目标：提升医生接诊效率，并增加基础处方和医嘱质量控制。

新增模型：

- `MedicalRecordTemplate`：病历模板。
- `CommonDiagnosis`：常用诊断。
- `CommonMedicalOrder`：常用医嘱。
- `PrescriptionTemplate`：常用处方模板。
- `PrescriptionTemplateItem`：模板药品明细。
- `PrescriptionReviewLog`：处方审核记录。

后端能力：

- 医生维护个人模板，管理员维护科室公共模板。
- 接诊时套用病历模板。
- 快速添加常用诊断和医嘱。
- 从处方模板生成处方。
- 药房审核通过或驳回处方。
- 驳回处方由医生修改后重新提交。
- 提交处方前做药品启用、库存、重复药品、必填字段等基础校验。

验收标准：

- 医生能用模板快速完成接诊。
- 药房能驳回处方并记录原因。
- 处方提交前有基础质量提示。
- 模板停用不影响历史病历。

## 阶段 6：患者端提醒和排队

目标：让患者端能看到下一步事项、排队进度和复诊入口。

新增模型：

- `PatientNotification`：患者站内通知。
- `QueueTicket`：排队票号。
- `DoctorQueueState`：医生队列状态。
- `FavoriteDoctor`：收藏医生。

后端能力：

- 预约成功生成通知。
- 就诊前生成预约提醒。
- 签到后生成排队票号。
- 医生开始接诊更新当前叫号。
- 患者查询排队进度和预计等待。
- 收藏/取消收藏医生。
- 从历史就诊记录发起复诊预约。

业务规则：

- 只有签到后进入排队。
- 取消、爽约、完成的挂号不能继续排队。
- 预计等待按候诊人数和平均接诊时长估算。
- 通知先做站内消息，不接短信或微信模板消息。

验收标准：

- 患者能看到预约提醒和排队进度。
- 医生能基于队列接诊。
- 通知不依赖外部平台。
- 排队状态和挂号状态一致。

## 阶段 7：运营看板升级

目标：把 Dashboard 从基础数量统计升级为门诊运营指挥台。

新增指标：

- 今日挂号数、签到数、接诊数、完成就诊数。
- 待收费订单数、待发药处方数。
- 到诊率、爽约率。
- 科室负载、医生接诊量排行。
- 平均等待时间。
- 收入趋势。
- 药品低库存和近效期预警。
- 停诊/改约数量。

后端接口：

- `/api/admin/dashboard/overview`。
- `/api/admin/dashboard/outpatient`。
- `/api/admin/dashboard/revenue`。
- `/api/admin/dashboard/pharmacy-alerts`。
- `/api/admin/dashboard/queue-pressure`。

实现原则：

- 初期用实时聚合查询。
- 聚合逻辑放 service 层。
- 后续数据量大时再引入统计快照表。
- 住院、医保、LIS、PACS 后续加入时扩展卡片，不重写看板结构。

验收标准：

- 管理员能看到今日业务压力。
- 能识别待处理事项。
- 能看到科室和医生负载。
- 能看到药房库存风险。
- 看板数据与业务列表口径一致。

## 阶段 8：住院管理

目标：增加基础住院闭环：入院、床位、住院医嘱、费用、出院。

新增模型：

- `Ward`：病区。
- `Bed`：床位。
- `InpatientAdmission`：住院登记。
- `BedAssignment`：床位分配。
- `InpatientMedicalRecord`：住院病历摘要。
- `InpatientOrder`：住院医嘱。
- `InpatientCharge`：住院费用明细。
- `DischargeRequest`：出院申请。

后端能力：

- 入院登记。
- 分配床位、换床、释放床位。
- 查看住院患者列表。
- 开立住院医嘱。
- 住院费用进入统一费用中心。
- 停止医嘱。
- 生成住院费用清单。
- 出院申请和出院模拟结算。
- 出院后释放床位。

状态规则：

- 住院：`PENDING -> ADMITTED -> DISCHARGE_REQUESTED -> DISCHARGED`。
- 床位：`AVAILABLE -> OCCUPIED -> CLEANING -> AVAILABLE`。
- 医嘱：`DRAFT -> ACTIVE -> STOPPED -> COMPLETED -> CANCELLED`。
- 出院：`REQUESTED -> APPROVED -> SETTLED -> COMPLETED`。

验收标准：

- 能完成入院到出院基础闭环。
- 床位状态准确。
- 住院费用进入统一费用中心。
- 患者端能查看住院摘要。

## 阶段 9：医保模拟结算

目标：演示自费/医保拆分、预结算、正式结算和撤销结算。

新增模型：

- `InsuranceProfile`：医保档案。
- `InsuranceCatalogMapping`：医保目录映射。
- `InsuranceSettlement`：医保结算单。
- `InsuranceSettlementItem`：医保结算明细。
- `InsuranceProviderLog`：医保模拟调用日志。

后端能力：

- 维护医保档案。
- 维护医保目录映射。
- 对门诊订单或住院费用执行预结算。
- 正式结算。
- 撤销结算。
- 退费时联动医保冲正。
- 通过 `InsuranceProvider` 调用 `MockInsuranceProvider`。

模拟规则：

- 甲类按比例报销。
- 乙类先自付一部分，再按比例报销。
- 丙类全自费。
- 起付线、封顶线作为配置项。
- 参保类型影响报销比例。

验收标准：

- 能演示门诊医保结算。
- 能演示住院医保结算。
- 能看到医保/自费金额拆分。
- 后续可替换真实医保 provider。

## 阶段 10：LIS 检验

目标：增加检验申请到报告发布的模拟 LIS 闭环。

新增模型：

- `LabTestItem`：检验项目。
- `LabRequest`：检验申请。
- `LabRequestItem`：检验申请明细。
- `LabSample`：标本。
- `LabResult`：检验结果。
- `LabReport`：检验报告。
- `LabProviderLog`：模拟 LIS 调用日志。

后端能力：

- 医生从门诊/住院开立检验医嘱。
- 检验医嘱生成申请单和费用。
- 检验科处理待采样、待接收、待录入、待审核。
- 采样生成条码。
- 接收标本。
- 录入结果。
- 审核并发布报告。
- 患者端查看已发布报告。
- Mock LIS provider 模拟结果回传。

状态规则：

- 检验申请：`REQUESTED -> SAMPLED -> RECEIVED -> RESULTED -> REVIEWED -> PUBLISHED`。
- 标本：`CREATED -> COLLECTED -> RECEIVED -> REJECTED`。
- 报告：`DRAFT -> REVIEWED -> PUBLISHED`。

验收标准：

- 门诊/住院医生都能开检验。
- 检验科能完成采样到报告发布。
- 患者端能查看已发布检验报告。
- 检验费用进入统一费用中心。

## 阶段 11：PACS 影像

目标：增加影像申请、预约、检查、报告和模拟影像查看。

新增模型：

- `ImagingExamItem`：影像检查项目。
- `ImagingRequest`：影像申请。
- `ImagingAppointment`：影像预约。
- `ImagingStudy`：影像检查记录。
- `ImagingReport`：影像报告。
- `PacsProviderLog`：模拟 PACS 调用日志。

后端能力：

- 医生从门诊/住院开影像申请。
- 影像申请生成费用明细。
- 影像科安排检查预约。
- 检查完成后生成模拟影像记录。
- 影像医生录入报告。
- 审核并发布报告。
- 医生端和患者端查看已发布报告。
- Mock PACS provider 生成影像链接或模拟文件路径。

状态规则：

- 影像申请：`REQUESTED -> SCHEDULED -> CHECKED_IN -> PERFORMED -> REPORTED -> REVIEWED -> PUBLISHED`。
- 影像预约：`SCHEDULED -> CHECKED_IN -> COMPLETED -> CANCELLED`。
- 影像报告：`DRAFT -> REVIEWED -> PUBLISHED`。

验收标准：

- 门诊/住院医生都能开影像申请。
- 影像科能完成预约到报告发布。
- 医生和患者能查看已发布影像报告。
- 影像费用进入统一费用中心。
- PACS 外部对接接口已预留，默认走模拟 provider。

## 风险与控制

- 范围过大：按 11 个阶段逐步实现，每阶段独立验收。
- 模型重复：患者、费用、医嘱、药品、报告、审计尽量共用基础能力。
- 抽象过度：第一阶段不先做复杂万能表单引擎，先用轻量配置和专用页面并存。
- 外部系统依赖：支付、医保、LIS、PACS 只做 mock provider，真实对接留接口。
- 业务状态混乱：所有状态动作进入 service 层，测试覆盖非法状态迁移。

## 总体验收标准

- 管理端可维护基础主数据。
- 门诊排班、预约、改约、签到、接诊、收费、发药、提醒、排队形成完整闭环。
- 支付和退款具备流水与审计。
- 药房库存约束真实参与发药。
- 医生端具备模板化诊疗和基础质量提示。
- 患者端能查看提醒、排队、费用、报告和住院信息。
- 运营看板能反映门诊压力、收入、库存和队列。
- 住院能完成入院、床位、医嘱、费用、出院基础闭环。
- 医保能演示预结算、正式结算、撤销和退费冲正。
- LIS 能完成检验申请、采样、结果、报告发布。
- PACS 能完成影像申请、预约、检查、报告和模拟影像查看。
- 所有外部系统默认使用 mock provider，并保留替换真实 provider 的接口边界。
